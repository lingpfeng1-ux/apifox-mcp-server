# Apifox MCP Server 安装使用指南

> 这是一个让 AI(Claude / Codex 等)直接操作 Apifox 项目的 MCP 服务:增删改查接口、数据模型、目录,支持多项目切换。v2.0.0,18 个工具。

## 一、前置准备

### 1. 装 Node.js(>= 20)
```bash
node -v   # 确认 >= 20
```
没装的话去 https://nodejs.org 下 LTS 版。

### 2. 获取 Apifox Access Token
1. 登录 Apifox 网页版 → 右上角头像 → **账号设置**
2. 左侧 **API 访问令牌** → **新建令牌**
3. 复制保存令牌(形如 `afxp_xxxxxxxx`)

> ⚠️ 令牌是个人令牌,权限边界由你在 Apifox 里的项目权限决定。建议用有目标项目写权限的账号生成。

### 3. 拿到项目 ID
打开你的 Apifox 项目,URL 里 `/project/{数字}` 那个数字就是项目 ID,例如 `https://app.apifox.com/project/8221779` → 项目 ID `8221779`。

## 二、安装(三步)

### 1. clone 并构建
```bash
git clone https://github.com/lingpfeng1-ux/apifox-mcp-server.git
cd apifox-mcp-server
npm install
npm run build
```
构建后产物在 `dist/index.js`。**记下这个目录的绝对路径**,下一步要用。

### 2. 配置你的 AI 客户端

#### Codex(本机 ~/.codex/config.toml)
```toml
[mcp_servers.apifox]
type = "stdio"
command = "node"
args = ['C:\\你\\clone\\的\\路径\\apifox-mcp-server\\dist\\index.js', "--project=8221779"]
startup_timeout_sec = 90

[mcp_servers.apifox.env]
APIFOX_ACCESS_TOKEN = "afxp_你的令牌"
```

#### Claude Desktop / Cursor / 其它 MCP 客户端
```jsonc
{
  "mcpServers": {
    "Apifox": {
      "command": "node",
      "args": [
        "/你/clone/的/路径/apifox-mcp-server/dist/index.js",
        "--project=8221779"
      ],
      "env": {
        "APIFOX_ACCESS_TOKEN": "afxp_你的令牌"
      }
    }
  }
}
```

> `--project` 是默认项目 ID(**可选**):不写也行,每次调用工具时传 `projectId` 指定;写了的话不传就用它。
> 路径用正斜杠 `/` 或双反斜杠 `\\`(JSON/TOML 里反斜杠要转义)。

### 3. 重启 AI 客户端
重启后让它连上 MCP。验证:问 AI「列出 Apifox 项目的模块」,它应该能调 `apifox_list_modules` 返回结果。

## 三、怎么用(给 AI 下指令的例子)

装好后,**你不用记工具名,直接用自然语言说要做的事,AI 会自己选工具**。几个典型场景:

### 场景 1:查/找接口
```
帮我看下项目 8221779 里 KAZ-PDP 模块下,路径含 "images" 的接口有哪些
```
AI 会用 `apifox_search_endpoints`(moduleName + keyword)精准检索,只返回匹配的几条,不会把几百个接口全拉进来。

### 场景 2:改某个接口
```
把项目 8221779 里 /activity/create 这个接口的描述改成「创建活动(新版)」
```
AI 流程:search 找到 apiId → get_endpoint 拿现有 → update_endpoint 改。你说"哪个接口+改什么"即可。

### 场景 3:改数据模型字段(重点,有同名坑)
```
把项目 8221779 里 /activity/create 请求体引用的那个模型,给它的 name 字段加 required
```
**关键**:项目里可能有多个同名模型,AI 必须用接口**实际引用的 id**(从 `get_endpoint(raw=true)` 的 `$ref: #/definitions/{id}` 取),再用 `update_schema(id, ...)` 按 id 精确改。AI 现在的工作流就是这样的,你只管描述要改哪个接口的哪个字段。

### 场景 4:建新接口 + 数据模型
```
在项目 8221779 的 KAZ-PDP 模块 Client-Image 目录下,建一个 POST /xxx 接口,
请求体引用一个新模型 XxxReq(含字段 a/b),响应引用 XxxResp
```
AI 会:create_schema 建模型 → create_endpoint 建接口(用 folderName 自动定位目录)。

### 场景 5:删东西(小心,有预览)
```
我想删掉项目 8221779 里 xxx 目录,先告诉我里面有哪些接口
```
AI 会用 `apifox_delete_folder({dryRun:true, moduleId})` 预览将删除的接口列表(含子目录),确认后再真删。**删除不可逆**。

## 四、多项目切换

每个工具都支持 `projectId` 参数,不传用默认(`--project`),传了就用指定的:
```
看下项目 7834388 的模块    // AI 自动传 projectId=7834388
```
所以一台机器配一个默认项目,临时操作别的项目直接在指令里说项目 ID 即可,不用改配置重启。

## 五、常见问题

### Q: 启动报「connection closed: initialize response」
**A**: 99% 是 config 里 `dist/index.js` 路径不对(目录改名/移动后没更新)。打开配置看 args 里的路径,确认文件真实存在:
```bash
ls /你/配的/路径/dist/index.js   # 必须能列出文件
```
不存在就改回正确的 clone 路径,重启客户端。

### Q: 报「端点返回 302 重定向」或「端点返回空响应」
**A**: 正常保护机制——某些 Apifox 旧端点对个人令牌不可用,服务会明确报错而不是假装成功。按错误提示换种操作(比如建目录用 import 而非直接 API)。

### Q: 改了数据模型但接口文档没变化
**A**: 大概率改到了同名模型的另一份。让 AI 用 `get_endpoint(raw=true)` 看接口 `$ref` 的精确 id,再用 `update_schema(那个id, ...)` 改。

### Q: 单元测试 / 集成测试怎么跑
```bash
npm test                 # 单元测试(无副作用)
npm run test:integration # 集成 smoke(真实 API,需配环境变量,默认 skip)
```

## 六、能力边界(个人令牌的限制)

个人令牌(`afxp_`)下:
- ✅ 接口:增删改查全可用
- ✅ 数据模型:读、建、按 id 精确改、删
- ✅ 目录:列表/定位(import+http-apis 关联重建)、删(递归删子目录)
- ⚠️ 目录无法直接"建空目录",只能通过 import 带标签的接口时附带创建
- 详细端点可用性见仓库 `docs/功能与实现方案.md`
