# Apifox MCP Server 功能与实现方案

版本：v2.0（重构后当前状态）
最后更新：2026-06-22

本文档说明该 MCP server **现在能做什么**,以及**每项能力是如何实现的**。

---

## 一、整体定位

这是一个把 Apifox 项目管理能力暴露给 AI/MCP 客户端(codex、Claude 等)的 stdio MCP server。
通过个人访问令牌(`afxp_` personal access token)操作 Apifox,支持在多个项目间切换。

核心约束:**personal token 只能访问 Apifox 的部分内部端点**,很多旧端点会返回 302 重定向
或 200 空 body。本服务的实现围绕"只用经过实测可用的端点,并把不可用的能力用可用端点重建"
这一原则展开。

---

## 二、能完成哪些功能(工具清单)

共 16 个工具,均支持可选 `projectId` 参数在调用时覆盖默认项目。

**上下文优化原则**:列表/搜索类工具只返回精简索引(不含 jsonSchema、大结构、内部噪音字段),
需要完整结构时用对应 `get_*` 工具。实测 `list_schemas` 从近 100KB 降到几百字符。

### 项目与模块
| 工具 | 功能 | 入参 |
|---|---|---|
| `apifox_get_project` | 获取项目详情 | projectId? |
| `apifox_list_modules` | 列出项目下的模块 | projectId? |

### 目录
| 工具 | 功能 | 入参 |
|---|---|---|
| `apifox_list_folders` | 列出指定模块下所有目录(moduleId/folderId/folderName/folderPath) | moduleName, projectId? |
| `apifox_find_folder` | 按模块名+目录名定位 moduleId 与 folderId | moduleName, folderName, projectId? |
| `apifox_delete_folder` | 删除接口目录 | folderId, projectId? |

### 接口(HTTP API)
| 工具 | 功能 | 入参 |
|---|---|---|
| `apifox_list_endpoints` | 列出接口索引(精简:id/name/method/path/folderId) | moduleId?, projectId? |
| `apifox_search_endpoints` | 按关键词/方法/目录检索接口(精简索引) | keyword?, method?, folderId?, moduleId?, limit?, projectId? |
| `apifox_get_endpoint` | 接口详情(默认精简字段,raw=true 全量) | apiId, raw?, projectId? |
| `apifox_create_endpoint` | 创建接口(带成功校验) | name, method, path, folderId?, description?, tags?, parameters?, requestBody?, responses?, projectId? |
| `apifox_update_endpoint` | 更新接口 | apiId, name?, method?, path?, description?, parameters?, requestBody?, responses?, projectId? |
| `apifox_delete_endpoint` | 删除接口(可选删除后回查) | apiId, verify?, projectId? |

### 数据模型
| 工具 | 功能 | 入参 |
|---|---|---|
| `apifox_list_schemas` | 列出模型索引(精简:id/name/folderId/description) | keyword?, moduleId?, limit?, projectId? |
| `apifox_get_schema` | 单个模型完整 jsonSchema(按 id 或名称) | idOrName, projectId? |
| `apifox_delete_schema` | 删除数据模型 | schemaId, projectId? |

### 导入导出
| 工具 | 功能 | 入参 |
|---|---|---|
| `apifox_import_openapi` | 导入 OpenAPI/Swagger(可同时创建数据模型) | spec, projectId? |
| `apifox_export_openapi` | 导出项目/模块为 OpenAPI | moduleId?, addFoldersToTags?, projectId? |

## 三、关键能力如何实现

### 1. 多项目切换

`projectId` 的解析在 `HttpClient.resolveProjectId()`,优先级:

```
调用参数 projectId > 启动参数 --project > 环境变量 APIFOX_PROJECT_ID > 抛错
```

每个能力方法都接受可选 `projectId`,内部解析后拼进 API 路径。`projectId` 只用于路由,
不会写进任何 create/update/import 的请求体。

### 2. 可用端点矩阵(实现的事实基础)

| 端点 | 方法 | 状态 | 用途 |
|---|---|---|---|
| `/api/v1/projects/{id}` | GET | 可用 | 项目详情 |
| `/api/v1/projects/{id}/modules` | GET | 可用 | 模块列表 |
| `/api/v1/projects/{id}/http-apis` | GET | 可用 | 接口列表(支持 `?moduleId`) |
| `/api/v1/projects/{id}/http-apis/{apiId}` | GET/POST/PUT/DELETE | 可用 | 接口 CRUD |
| `/api/v1/projects/{id}/export-openapi` | POST | 可用 | 导出(顶层 `moduleId` + scope ALL + `addFoldersToTags`) |
| `/api/v1/projects/{id}/import-data` | POST | 可用 | 导入(`{data, importFormat}`) |
| `/api/v1/projects/{id}/api-tree` | GET | 失效(200 空) | — |
| `/api/v1/projects/{id}/api-tree-folders` | POST/DELETE | 失效(302) | — |
| `/api/v1/projects/{id}/data-schemas` | GET | 可用 | 数据模型列表(只读) |
| `/api/v1/projects/{id}/data-schemas` | POST/PUT/PATCH/DELETE | 失效(302) | 模型写端点对 token 不开放 |
| `/api/v1/projects/{id}/schemas` | GET/POST | 失效(空/302) | 旧端点名,已弃用 |
| `/api/v1/projects/{id}/export-openapi` | GET | 失效(200 空) | — |

统一版本头：`X-Apifox-Api-Version: 2024-03-28`。

### 3. HTTP 层防"假成功"(核心修复)

`apifox/http.ts` 的 `HttpClient`:
- `maxRedirects: 0` + `validateStatus: () => true`：不跟随重定向、自行判定状态码。
- **3xx** → 抛"端点返回重定向,可能对当前 token 不可用"。
- **4xx/5xx** → 解析 Apifox 错误体(errorCode/errorMessage)抛 `ApifoxError`。
- **200 但 body 为空** → 抛"端点返回空响应",而不是返回空数据。

这样彻底杜绝旧实现里"`/api-tree-folders` 返回 302 却被当成 success=true"的假成功问题。

### 4. 目录能力 = 三段数据关联重建

由于 `/api-tree`、`/api-tree-folders` 不可用,目录信息无法直接获取。
`apifox/folders.ts` 用可用端点关联重建：

```
1. /modules                          → 模块名 → moduleId
2. POST /export-openapi(按 moduleId, addFoldersToTags)
                                      → 每个 method+path → 目录路径(x-apifox-folder)
3. /http-apis?moduleId               → 每个 method+path → folderId
以 "METHOD + path" 为连接键,得到 目录名/目录路径 ↔ folderId
```

- `find_folder`：解析模块 → 关联 → 返回匹配目录的 folderId（目录名精确匹配,多级路径按最后一段匹配,模块名去空白/忽略大小写兜底）。
- `list_folders`：返回该模块下所有目录的 `{moduleId, moduleName, folderId, folderName, folderPath}`。

### 5. 接口 CRUD + 成功校验

`apifox/endpoints.ts`：
- `create`：POST `/http-apis`，**校验返回含真实 `id`**,否则抛错。
- `update`：PUT `/http-apis/{apiId}`,patch 字段透传(method 自动大写)。
- `delete`：DELETE `/http-apis/{apiId}`,`verify=true` 时删除后回查(404 视为删除成功)。

### 6. 接口的参数/请求体/响应透传

`create_endpoint` / `update_endpoint` 接受 `parameters` / `requestBody` / `responses`,
原样透传给 Apifox（实测 PUT 真实生效)。
- `parameters` 形如 `{ path:[], query:[], header:[], cookie:[] }`。
- 推荐用法：先 `get_endpoint` 拿现有结构,改完整体回传,避免字段格式出错。

### 7. 数据模型工作流（读用 data-schemas,建/改用 import）

数据模型的 `data-schemas` 写端点(POST/PUT/PATCH/DELETE)对 personal token 返回 302,
但读/建/改都能用可用端点完成(覆盖参数由逆向 Apifox 客户端得到):
- 读:`apifox_list_schemas`(`GET /data-schemas`)。
- 建:`import_openapi`——`components.schemas` 定义模型,`paths` 用 `$ref` 引用。
- 改字段:`import_openapi` 重导同名模型并传 **`schemaOverwriteMode:"name"`**(默认 `ignore` 不覆盖,
  这是之前一直改不动的根因);改接口同理用 `apiOverwriteMode:"methodAndPath"`。
- 删:仍需 Apifox UI(端点 302)。

`SchemaOverwriteModeEnum` 值:`ignore` / `name` / `nameAndFolder` / `merge`;
`ApiOverwriteModeEnum` 值:`ignore` / `methodAndPath` / `methodAndPathAndFolder` / `merge`。
import 统计校验同时计入 `schemaCollection` 的 create/update,避免"仅改模型"被误判为失败。

`apifox/importExport.ts` 的 `importOpenAPI`:
- 请求体使用正确格式 `{ data: <spec字符串>, importFormat: 'openapi' }`
  （旧实现的 `{input, inputType, options}` 实际不生效,createCount 恒为 0,已废弃）。
- 校验导入统计：若接口与目录创建/更新数全为 0,抛错,避免"导入成功但啥都没建"。

## 四、架构分层

```
src/
  index.ts            入口:启动 MCP server、注册 list/call handler、统一错误包装
  config.ts           配置解析(token / 默认 projectId / baseURL / api 版本)
  errors.ts           统一错误类型 ApifoxError
  apifox/
    http.ts           底层 HTTP:鉴权、302/空 body 识别、错误归一化、resolveProjectId
    projects.ts       项目 / 模块能力 + 按名解析模块
    endpoints.ts      接口 CRUD(写操作带成功校验)
    folders.ts        目录能力(export + http-apis 关联重建)
    importExport.ts   import-data 导入 / export-openapi 导出
    types.ts          领域类型
    index.ts          能力层门面 Apifox(装配各 Service)
  tools/
    registry.ts       MCP 工具注册表(schema + handler,薄封装调能力层)
__tests__/            vitest 单元测试 + opt-in 集成 smoke
docs/                 文档
```

数据流：`MCP 请求 → index.ts 分发 → tools/registry handler → apifox 能力层 Service → http.ts → Apifox API`。
错误一律以 `ApifoxError` 上抛,在 `index.ts` 统一包成 MCP 的 `isError` 响应。

## 五、已知限制

- **目录只能建不能删/改**:`/api-tree-folders` 写端点 302,目录靠 import 附带创建。
- **数据模型可读、可建、可改字段**:读用 `list_schemas`,建/改字段用 `import_openapi`
  (改字段传 `schemaOverwriteMode:"name"`);仅**删除**需 Apifox UI(`/data-schemas` 删除端点 302)。
  建错了需在 Apifox UI 手动删。
- **目录无法直接创建空目录**：建目录依赖 `import_openapi` 导入带 tag 的接口时附带创建。
- **接口更新的复杂结构需调用方自行保证格式**：parameters/requestBody/responses 原样透传,
  建议先 get 再改。
- 以上限制均源于 personal token 的端点可见性,非本服务实现缺陷。

## 六、测试

- 单元测试(`npm test`)：mock HTTP,覆盖配置解析、302/空 body 识别、projectId 优先级、
  目录关联逻辑、create 成功校验、import 统计校验、复杂字段透传。无副作用。
- opt-in 集成 smoke(`npm run test:integration`)：真实打 Apifox API,需环境变量开关；
  写操作(create/update/delete)再受 `APIFOX_RUN_WRITE=1` 二次开关控制,自带清理。
